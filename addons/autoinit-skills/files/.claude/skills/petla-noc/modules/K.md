# Moduł K — MODERNIZACJA SKŁADNI + ŻYWA DOKUMENTACJA (wypełniacz, najniższy priorytet)

Dwie części: (K1) var→const/let i równoważne zmiany zero-behavioral — za bramką,
osobny commit per plik; (K2) auto-aktualizacja ARCHITECTURE.md i CHANGELOG.md.
Cały moduł WYŁĄCZONY w RED MODE (lista usera).

## K1. var → const/let (zero-behavioral)

1. Bramka pełna per plik (SKILL.md): testy green przed/po, branch, nie-RED.
   Plik bez testów → POMIŃ (nawet "oczywisty" — zero-behavioral bez dowodu
   testowego to oksymoron w 20k-liniowym vibekodzie).
2. Zakres zmian DOZWOLONYCH (nic ponad to):
   - `var` → `const` (brak reasygnacji w scope) / `let` (jest reasygnacja);
   - UWAGA pułapki GAS/ES5: var ma function-scope — var w bloku używany poza
     blokiem → ZOSTAW (zmiana na let zmieniłaby semantykę!); var hoisting
     (użycie przed deklaracją) → ZOSTAW + raport; duplikaty `var x` w jednej
     funkcji → scal ostrożnie lub ZOSTAW.
   - NIE: arrow functions, template literals, destrukturyzacja, ===, formatowanie
     — to NIE jest zero-behavioral refaktor składni deklaracji (poza zakresem K).
3. Mechanika: per plik → zmiany → harness green → commit `syntax-modern`
   (1 plik = 1 commit; message wylicza ile var zamieniono) → następny plik.
   Fail → rollback pliku + raport ("plik oporny — var z pułapką scope").

## K2. Żywa dokumentacja (nie wymaga bramki testowej — nowe pliki .md)

1. `<projekt>/ARCHITECTURE.md` — GENERUJ z map.json (moduł A; mapa świeża z tej
   nocy) + danych G/H:
   - Entry pointy (handlery + cele triggerów/menu/gsrun) z 1-zdaniowym opisem (JSDoc),
   - Triggery (skąd instalowane, co wołają),
   - Przepływ danych: które funkcje czytają/piszą które arkusze (uses_gas_api +
     getSheetByName z kodu), webhooki/maile (z H),
   - Diagram zależności głównych modułów (mermaid flowchart, max 25 węzłów —
     najczęściej wołane; pełnia jest w map.json).
   Nagłówek pliku: `<!-- AUTO-GENERATED by petla-noc <data> from map.json — edycje ręczne zostaną nadpisane -->`.
2. `<projekt>/CHANGELOG.md` — sekcja per noc: `## cleanup/<data>` + lista commitów
   sesji (`git log --oneline <session_start_commit>..HEAD`) pogrupowana po
   kategoriach (fix/quarantine/tests/...). Dopisuj na GÓRĘ pliku; istniejące
   sekcje nietknięte.
3. Commit `docs` (oba pliki razem, per projekt).
